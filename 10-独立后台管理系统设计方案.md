# 愿望卡（WishCard）独立后台管理系统设计方案

## 一、背景与目标

当前「愿望卡」所有管理操作（邀请码、用户数据、反馈等）都散落在前端应用中，存在以下问题：

1. **邀请码由各家庭管理员自行管理**，无法全局管控，内测阶段无法有效控制注册入口
2. **没有全局视角**，无法查看系统整体运营数据（用户量、活跃度、任务完成量等）
3. **用户反馈无法集中处理**，当前反馈仅前端 alert，未持久化
4. **缺乏系统级配置能力**，功能开关、模板管理、通知推送等均无法远程控制
5. **数据安全风险**，任何登录用户理论上可通过 API 访问非本家庭数据

**目标**：建立完全独立的后台管理系统，与前端应用分离部署，仅限系统管理员访问，实现全局运营管控。

---

## 二、系统架构

```
┌─────────────────────┐     ┌─────────────────────┐
│   前端应用 (用户端)    │     │   后台管理系统 (Admin) │
│   wishcard.app       │     │   admin.wishcard.app │
│   React + Vite       │     │   React + Vite       │
└──────────┬──────────┘     └──────────┬──────────┘
           │                            │
           │        Supabase            │
           │   ┌──────────────────┐     │
           └───►│  数据库 (Postgres) │◄────┘
               │  Auth / Storage  │
               │  Edge Functions  │
               └──────────────────┘
```

**关键设计**：
- 后台为独立 React 项目，独立域名部署（如 `admin.wishcard.app`）
- 共用同一个 Supabase 实例，通过 RLS + 自定义 Claims 区分权限
- 后台管理员通过 Supabase Auth 的 `app_metadata.admin = true` 鉴权
- 数据库新增 `admin_users` 表记录系统管理员

## 三、功能模块详细设计

### 模块 1：仪表盘（Dashboard）

**目的**：一屏掌握系统核心运营数据

| 数据项 | 说明 |
|--------|------|
| 总注册用户数 | 按日/周/月趋势图 |
| 活跃用户数 (DAU/MAU) | 基于最近登录/操作时间 |
| 家庭总数 | 含访客模式家庭 |
| 今日新增注册 | 含注册来源（邮箱/手机/游客） |
| 任务完成总数 | 按日趋势 |
| 星星流通总量 | 发放 vs 消耗 |
| 邀请码使用率 | 已用/已发放 |
| 用户反馈待处理数 | 未读反馈条数 |

**图表**：折线图（用户增长）、饼图（注册来源）、柱状图（任务完成趋势）

---

### 模块 2：用户管理

**目的**：全局管理所有注册用户

| 功能 | 说明 |
|------|------|
| 用户列表 | 分页展示，支持搜索（邮箱/手机/昵称/ID） |
| 用户详情 | 查看用户基本信息、所属家庭、成员关系、操作日志 |
| 封禁/解封 | 封禁后用户无法登录，数据保留 |
| 重置密码 | 管理员可重置用户密码 |
| 删除用户 | 软删除，关联数据清理（家庭、成员、任务等） |
| 角色管理 | 查看并调整用户角色（普通用户/系统管理员） |
| 设备管理 | 查看用户登录设备、强制下线 |

**数据来源**：`auth.users` + `members` + `families`

---

### 模块 3：家庭管理

**目的**：查看和管理所有家庭数据

| 功能 | 说明 |
|------|------|
| 家庭列表 | 分页展示，支持搜索（家庭名/ID/邀请码） |
| 家庭详情 | 查看成员列表、任务统计、星星流通、活跃度 |
| 强制解散 | 违规家庭强制解散，通知所有成员 |
| 数据修正 | 管理员可修正异常数据（星星数、任务状态等） |
| 家庭邀请码 | 查看和管理家庭邀请码（区别于注册邀请码） |

**数据来源**：`families` + `members`

---

### 模块 4：邀请码管理（核心）

**目的**：全局管理注册邀请码，控制内测注册入口

| 功能 | 说明 |
|------|------|
| 邀请码列表 | 全部邀请码，支持搜索/筛选（状态/创建时间/使用情况） |
| 创建邀请码 | 单个创建 / 批量生成（指定数量、前缀、使用次数、有效期） |
| 启用/停用 | 一键切换邀请码状态 |
| 删除邀请码 | 删除未使用的邀请码 |
| 使用记录 | 查看每个邀请码的使用详情（谁、何时使用） |
| 统计分析 | 邀请码使用率、转化率 |

**新增数据库表**：`invite_code_usages`（记录每次使用详情）

```sql
CREATE TABLE invite_code_usages (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  invite_code_id uuid REFERENCES invite_codes(id),
  user_id uuid REFERENCES auth.users(id),
  used_at timestamptz DEFAULT now()
);
```

### 模块 5：内容管理

**目的**：管理应用内展示的内容资源

| 子模块 | 说明 |
|--------|------|
| **任务模板库** | 管理全局任务模板（`templates.ts` 中的数据迁入数据库），支持增删改、分类、排序 |
| **奖励模板库** | 管理全局奖励模板，与任务模板类似 |

| **公告管理** | 创建系统公告，推送至用户端（弹窗/通知栏） |
| **引导页管理** | 管理首次启动的引导页内容 |

**新增数据库表**：
- `task_templates`（全局任务模板）
- `reward_templates`（全局奖励模板）
- `announcements`（系统公告）
- `app_config`（应用配置项）

### 模块 6：反馈管理

**目的**：集中处理用户反馈

| 功能 | 说明 |
|------|------|
| 反馈列表 | 按状态/分类/时间筛选 |
| 反馈详情 | 查看反馈内容、截图、联系方式 |
| 回复反馈 | 管理员直接回复，用户端可查看 |
| 标记处理 | 已读/处理中/已解决/已关闭 |
| 统计 | 按分类统计反馈数量和趋势 |

**新增数据库表**：`feedbacks`

```sql
CREATE TABLE feedbacks (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid REFERENCES auth.users(id),
  family_id uuid REFERENCES families(id),
  category text NOT NULL,
  content text NOT NULL,
  images jsonb DEFAULT '[]',
  contact text,
  status text DEFAULT 'pending',  -- pending/processing/resolved/closed
  admin_reply text,
  replied_by uuid,
  replied_at timestamptz,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);
```

## 四、权限设计

### 4.1 系统管理员角色

| 角色 | 权限 |
|------|------|
| **超级管理员** | 所有权限 + 管理员账号管理 |
| **运营管理员** | 仪表盘、用户管理、家庭管理、邀请码、反馈、内容管理 |
| **客服管理员** | 反馈管理、用户查看（只读） |

### 4.2 实现方式

```sql
-- 新增管理员表
CREATE TABLE admin_users (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid REFERENCES auth.users(id) UNIQUE,
  role text NOT NULL DEFAULT 'operator',  -- super_admin / operator / support
  created_at timestamptz DEFAULT now()
);

-- 设置用户 admin claim（通过 Supabase Edge Function 或 SQL）
-- auth.users.app_metadata: { "admin": true, "admin_role": "super_admin" }
```

**RLS 策略**：
- 后台 API 请求必须携带 `admin` claim
- 非 admin 用户请求后台数据时返回 403
- 前端应用保留原有 RLS（只能访问本家庭数据）

```
│   │   ├── Settings.tsx        # 系统配置
│   │   └── AuditLogs.tsx       # 操作日志

│   │   ├── Sidebar.tsx         # 侧边导航

│   │   ├── api.ts              # 后台 API 封装

│   └── App.tsx

└── vercel.json
```

### 5.2 技术栈

| 类别 | 选型 | 说明 |
|------|------|------|
| 框架 | React 18 + Vite | 与前端应用一致 |
| UI 库 | Tailwind CSS + shadcn/ui | 后台风格，专业高效 |
| 图表 | Recharts | 轻量级图表库 |
| 路由 | React Router v6 | |
| 数据层 | Supabase JS Client | 共用同一实例 |
| 部署 | Vercel | 独立域名 |

### 5.3 鉴权流程

```
1. 管理员访问 admin.wishcard.app
2. 输入邮箱+密码登录
3. Supabase Auth 验证 → 检查 app_metadata.admin === true
4. 通过 → 进入后台；未通过 → 提示无权限
5. 所有 API 请求携带 JWT，RLS 校验 admin claim
```

## 七、实施优先级

### P0 - 第一期（核心，立即实施）

1. **独立后台项目搭建** - 新建 `wishcard-admin` 项目
2. **管理员鉴权** - 登录页 + admin claim 校验
3. **邀请码管理** - 从前端应用移至后台，全局管控
4. **仪表盘** - 核心数据概览
5. **用户管理** - 列表/搜索/封禁

### P1 - 第二期（1-2周内）

6. **家庭管理** - 列表/详情/数据修正
7. **反馈管理** - 前端反馈持久化 + 后台处理
8. **系统配置** - 远程开关和参数
9. **内容管理** - 任务/奖励模板迁入数据库

### P2 - 第三期（运营成熟后）

10. **数据分析** - 深度分析报表
11. **操作日志** - 审计追踪
12. **公告管理** - 推送能力
13. **多角色权限** - 细粒度权限控制

4. **增加** 公告展示逻辑（从 `announcements` 表读取）
5. **增加** 维护模式检测
6. **增加** 版本更新提示
